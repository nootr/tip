Who it helps

For teams that need trust to outlive one account or API.

TIP gives public trust evidence a portable format: signed, revocable, cacheable, and verifiable by clients.

01

Open-source maintainers

Publish and consume evidence about maintainership, reviews, releases, sponsorship, or incident response across forges and tools.

02

Communities and teams

Represent membership, roles, endorsements, and revocations without making a single chat server or spreadsheet authoritative.

03

Platforms and marketplaces

Ingest external trust evidence while keeping policy local instead of inventing another opaque global reputation score.

04

AI agents and automation

Check signed evidence about identities, capabilities, permissions, and revocations before automated systems take action.

Use TIP when

Positive claims are not enough.

TIP is built for public evidence that needs context, portability, and revocation.

Claims must verify offlineEvents carry signatures and stable IDs.
Revocations matterSafety evidence can override stale positive claims.
Trust policy is localApps decide which issuers and claims count.
Nodes are not authoritiesThey cache and transport evidence clients verify.

Protocol model

Trust is evidence, not a score.

TIP keeps the base layer small: keys sign events, nodes store valid public evidence, and clients choose their own trust policy.

01

Cryptographic identity

The canonical identity is an Ed25519 public key. Handles, domains, and usernames are claims.

02

Signed event log

Claims, attestations, and revocations are append-only signed events with stable SHA-256 IDs.

03

Local trust policy

Every app decides which issuers, claims, and attestations matter in its own context.

Anti-positioning

TIP is deliberately not a trust oracle.

Not OAuth

TIP does not log users in or grant sessions. It publishes verifiable evidence.

Not KYC

TIP does not prove real-world identity. It lets issuers sign claims and attestations.

Not blockchain

No token, mining, staking, or consensus network is required for TIP 0.1.

Not a global score

Clients apply their own policy. Trust remains contextual and auditable.

How it moves

From key to portable evidence.

  1. Generate key

    Create a local Ed25519 identity.

  2. Publish claims

    Sign handles, links, or public statements.

  3. Receive attestations

    Other issuers sign statements about your key.

  4. Sync nodes

    Explicit peers exchange valid events with cursor pagination.

Reference node

Run a small HTTP node for your community or product.

The Rust node stores events in SQLite, exposes a compact HTTP API, and can periodically pull from explicitly configured peers. Run one as a community registry, public evidence mirror, marketplace signal cache, or local trust-policy backend.

Node config

[node]
bind = "127.0.0.1:8080"
db = "tip-node.sqlite3"
key = "tip-node-key.json"
name = "Local TIP Node"
description = "Community trust registry"
website = "https://example.com"
contact = "mailto:admin@example.com"

[sync]
on_start = true
limit = 500
periodic_seconds = 300
full_resync_seconds = 86400

[[peers.nodes]]
url = "http://127.0.0.1:8081"
expected_node_public_key = "peer-node-public-key"
name = "Community peer"

Quick start

Create, submit, and query trust events.

cargo install --path crates/tip-cli
cargo install --path crates/tip-node

cp tip-node.example.toml tip-node.toml
tip-node serve --config tip-node.toml

tip key generate --name default
tip identity create --out identity.json
tip claim add github joris --out github-claim.json

tip event submit-batch identity.json github-claim.json \
  --node http://127.0.0.1:8080

tip query --subject <public-key> --limit 100 \
  --node http://127.0.0.1:8080

Build with it

Start with one trust problem.

Pick a community, project, marketplace, or agent workflow where signed claims and revocations would remove ambiguity. Run a node, publish events, and keep policy explicit.
